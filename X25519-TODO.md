There are a couple of issues with X25519 on SymCrypt (Windows, SymCrypt-OpenSSL). X25519 on OpenSSL and macOS works
as expected.

There are essentially four problems with SymCrypt Curve25519 when turning it in to X25519.

1. Private Key adjustment (status: fixed ✅). RFC 7748 states:

    > Scalars are assumed to be randomly generated bytes.  For X25519, in
    > order to decode 32 random bytes as an integer scalar, set the three
    > least significant bits of the first byte and the most significant bit
    > of the last to zero, set the second most significant bit of the last
    > byte to 1 and, finally, decode as little-endian.  This means that the
    > resulting integer is of the form 2^254 plus eight times a value
    > between 0 and 2^251 - 1 (inclusive).

    Windows does not do this. If you give it a private key that does _not_ fit this conformance, the private key is
    rejected. OpenSSL and macOS do this adjustment for you at import time.

    To work around this we can take the same approach that SymCrypt-OpenSSL does when they use SymCrypt Curve25519 to
    provide X25519: adjust the key for you, an keep track of the adjusted bits so that re-exporting the private key
    round-trips correctly. Since the adjusted bits do not have security value (they are coerced in to a certain state)
    we can store just these non-security providing bits without storing the entire private key.

2. Low-order value public keys (status: fixable ⚠️)

    X25519 is designed such that it does not need to "validate" keys provided the private key conform to the key adjustments.
    That means "off curve" or on-twist keys are acceptable to use, whereas with NIST curves, off-curve keys can present
    a real problem. The corresponding public key in SymCrypt validates sub-group membership via [GOrd]*P == O.

    This is "unnecceary" with X25519. Both OpenSSL and CryptoKit do not do this validation for X25519 because it is designed
    to withstand this problem due to its cofactor. SymCrypt does this validation anyway - it is unclear if this is intentional
    or if it it inherited it as a default check.

    For Windows, this can be fixed by passing `BCRYPT_NO_KEY_VALIDATION` to `BCryptImportKeyPair`. To my knowledge, this
    is the only validation that happens on X25519 public keys, so we won't be turning off other validations, either.

    For SymCrypt-OpenSSL - there is no easy fix. We would need SymCrypt-OpenSSL to pass `SYMCRYPT_FLAG_KEY_MINIMAL_VALIDATION`
    to SymCrypt, or expose a setting of some kind.

    Impact of not fixing: There are _twenty_ keys that will fail subgroup order validation. Statistically they are extremely
    unlikely to exist. Most will appear to be "obviously" not random but a few like `e0eb7a7c3b41b8ae1656e3faf19fc46ada098deb9c32b1fd866205165f49b800`
    where [8]P = O but [4]P != O will appear to be random, but not import because they are low order.

    What the spec says: The spec says these keys should be permitted, and Wycheproof tests exist for these keys.

3. Non-canonical public keys  (status: fixable ⚠️)

    RFC 7748 states

    > Implementations MUST accept non-canonical values and process them as
    > if they had been reduced modulo the field prime.  The non-canonical
    > values are 2^255 - 19 through 2^255 - 1 for X25519

    Windows does not accept these non-canonical values. This is trivial to fix by simply flipping some bits and zeroing
    some components, which will effectively and efficiently compute u % field.

    However, all but two of these keys are going to result in being low order or identity keys. There is practically
    no value in implementing this without solving the low-order value problem, first.

    This is fixable in both SymCrypt-OpenSSL and Windows, but the SCOSSL fix has reduced value without the order validation
    fix, since it is not available there.

4. Identity keys (status: unfixable ❌, practically, not a concern)

    X25519 has one identity public key (all zeros) and a non-canonical representation of it. These keys will not import
    with SymCrypt. However, this is not a problem. Due to X25519 key clamping, there is no corresponding private key that
    will have the identity coordindate as its public key. This public key cannot occur naturally, not without a "dishonest"
    private key that was incorrect clamped. No implementation will generate a key pair with this as a valid public key.

    However, some implementations (CryptoKit, OpenSSL) will accept these as public keys anyway. This may be an acceptable
    divergence between platforms, or, we can block identity points at import time on all platforms.
